# かんべblog — 執筆スタイルガイド

## 基本コンセプト

**語り手として、読者に「紹介する」感覚で書く。**

一人で考えているんじゃなくて、隣に座っている人に話しかけるような文章。
フレンドリーだけど、物事を真剣に考えている人間の言葉。

---

## 口調のルール

### 敬語とタメ口の使い分け

**敬語を使う場面**
- 読者への問いかけ、導入部分
- 大事なことを伝えるとき
- 締めくくり

**タメ口を使う場面**
- 自分の体験・気持ちを語るとき
- 驚き・発見を共有するとき
- 読者との距離を縮めたいとき

### 具体例

```
❌ 悪い例（ずっと敬語）
「広尾町に移住して1年が経ちました。移住にはさまざまな困難がありましたが、
地域の方々のご支援により充実した生活を送ることができています。」

❌ 悪い例（ずっとタメ口）
「移住して1年。まじでいろいろあったけど最終的にはよかった。広尾めっちゃいい。」

✅ 良い例（ミックス）
「広尾町に来て、もう1年が経ちました。
正直、最初の3ヶ月は結構しんどかった。でも今は——これが正解だったと思ってます。
その理由を、ちゃんと話させてください。」
```

## 記事を読ませる技術

### 1. 冒頭で「つかむ」

最初の3行で読者を引き込む。問い・驚き・矛盾から始める。

```
✅ パターン例

【問い型】
「なぜ東京を捨てて、人口6,000人の町を選んだのか。
自分でも、うまく説明できなかった時期がある。」

【驚き型】
「AIが生成したJSONが、毎回微妙に壊れてた。
最初は自分のコードのせいだと思ってた。違った。」

【矛盾型】
「ゲストハウスをやりながら、AIを開発してる。
この組み合わせ、実は全然矛盾してないんです。」
```

### 2. 「見出し」は会話の続きとして書く

見出しも文章の一部。読者が「次を読みたい」と思う引きを作る。

```
❌ 悪い例
## 課題について
## 解決策

✅ 良い例
## そこで気づいたこと
## だから、こうした
```

### 3. 短い文と長い文を交互に

リズムが大事。

```
✅ 例
「試してみた。うまくいかなかった。
それでも、やめようとは思わなかったんです。なぜなら——
この問題の向こうに、絶対に面白いものがあると確信してたから。」
```

### 4. 読者への「語りかけ」を入れる

たまに直接話しかける。

```
「あなたが移住を考えているなら、これは読んでほしい。」
「わかります。最初は私もそう思ってました。」
「ちょっと想像してみてください。」
```

### 5. 締めくくりは「次」につなげる

読み終わった後に、何かが残るようにする。

```
✅ 例
「今日の話はここまでにします。
ただ、これはまだ途中の話で——続きは次の記事で書くつもりです。
気になる方は、ぜひまた来てください。」
```

## 構成テンプレート

```
1. 冒頭フック（3行以内で引き込む）
2. 背景・状況の説明（「こういう状況だった」）
3. 本題（体験・発見・考察）
   - 具体的なエピソードを中心に
   - 数字・固有名詞を使って「リアル」にする
4. 学んだこと・気づき
5. 締め（次につながる余韻）
```

## セキュリティ・プライバシールール（厳守）

### 絶対に記事に書かないもの

- **APIキー・トークン・パスワード** — たとえ「例として」でも書かない
- **自宅住所・電話番号** — 町名・地域名までにとどめる
- **他人のフルネーム** — 「地元の〇〇さん」「友人のエンジニア」のように
- **個人が特定できる写真の説明** — 他者が写る場合は必ず許可確認
- **未公開の事業情報・契約内容**

### コード例を書くときのルール

```
❌ 悪い例
OPENAI_API_KEY=sk-proj-xxxxxxxxxxxxxxxxxxxx
DB_PASSWORD=mypassword123

✅ 良い例
OPENAI_API_KEY=your_api_key_here
DB_PASSWORD=your_password_here
```

- `.env` ファイルの中身は絶対に載せない
- コード例には必ずプレースホルダーを使う
- 実際のURLにはダミーを使う（`example.com` など）

### 外部サービス・ツールへの言及

- セキュリティ上の問題があるツールには注記を入れる
- 「自己責任で」という免責を必要に応じて添える
